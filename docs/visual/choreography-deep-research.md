# Motion Principles (A1–A3)

- **Attention/Glance pulse:** Use very short, “popping” animations (≈ 70–150 ms). For example, IBM Carbon’s **micro-interaction** durations are 70–110 ms. A quick ease-out curve like Carbon’s _productive_ easing (`cubic-bezier(0.2,0,0.38,0.9)`) or a simple ease-out works well. (Apple toggles and switches also use ~100 ms for feedback.)
- **Working Loop (sustained activity):** Animate at a slower, steady pace—on the order of ~500–1000 ms per cycle. Longer loops should use gentle, smooth easing (ease-in-out or linear) so motion feels continuous without jarring. For example, spinners or loading indicators often rotate once per second. NN/g suggests “complex transitions” up to 400–500 ms, but for a looping action, even ~1 s (1000 ms) is common, using a soft sine/ease-in-out to avoid abrupt repeats.
- **Settle-to-Rest:** When stopping an activity, use a modest duration (≈ 200–300 ms) with a deceleration (ease-out) so motion gently comes to rest. NN/g notes that “substantial” movements feel good around 200–300 ms. Carbon’s _exit_ easing (`cubic-bezier(0,0,0.38,0.9)`) provides a natural slow-out. For example, fade-out or node reposition animations around 250 ms with ease-out feel calm and final.

**Staging/Staggering (A2):** Don’t animate 100 nodes at once. Sequence related changes in a choreographed way. For example, stagger new nodes/edges by ~20–50 ms each so that a cloud of appearing elements unfolds as a single “wave” rather than a blizzard. IBM specifically recommends a 20 ms offset per item (keeping the whole sequence ≤500 ms) to reduce cognitive load. More broadly, group animations and delay slightly between groups so the user can follow the order; e.g. reveal parent before children, or left-to-right. Finish each sequence by highlighting the final target (so the “last action” stands out).

**Cognitive Load (A3):** Humans track only a handful of moving objects at once. As a guideline, try to animate only a few elements together (e.g. under 5–8 simultaneous highlights). Too many moving parts overwhelm the viewer. If lots of changes occur, aggregate them or show only the “most recent N.” For example, highlight the 5 newest or most relevant nodes, and gray out or dim older ones. (This mimics how performance charts often only show the last few activity bursts.) In practice, limit concurrent animations; use grouping and dimming. IBM notes that staggering reduces cognitive load. Similarly, UI blogs warn that “too many animated elements overwhelm users”. Keeping most of the graph static while a few nodes pulse or edges animate is less fatiguing than blitzing everything.

# Live-Flow Visualizations (B4–B6)

- **Tracing/Profiler UIs:** Common distributed-tracing tools (Jaeger, Datadog, Honeycomb) use time-based flame graphs and static DAGs rather than fully animated flows. Datadog, for instance, shows **flame charts** (nested spans along a time axis) to display execution time. Datadog’s Service Map draws a live dependency graph where nodes (services) cluster by call volume, and you can highlight a service to see its upstream/downstream. Jaeger similarly offers **service dependency graphs** (a directed graph of service calls) and a span list. These UIs use color and size coding (e.g. thicker arrows for heavy call volume) but generally _static_ highlighting (hover or filter to focus). The emphasis is on labeling direction (arrows), per-span timelines, and the ability to drill into a path.
- **Game/Flow Analogies:** In game UIs, “telegraphing” an action often uses ghost trajectories or highlight rings. In dataflows, edges often carry directional cues: e.g. arrowheads, or nodes that “light up” sequentially. For example, the open-source SYNAPSE project uses glowing nodes with “drifting particles” and animated edges to show active traversal. In data/ETL tools, edges often have little arrows or moving dots to show flow direction (the React Flow ETL demo explicitly mentions _“directional edges: arrows showing data flow”_ and animated flows). Legible cues include clear arrowheads, consistent motion direction (particles always move source→target), and labels or color intensity that encode throughput or latency.
- **Particle Flow (B5):** Animating particles or “comets” along links can beautifully encode flow. React-Force-Graph even supports setting a number of moving particles per link, with tunable speed. Keep particles small and spaced so they look like a stream. Use their **direction** (particle motion) or add arrowheads to disambiguate flow. Speed can encode magnitude (more particles or faster speed = more traffic), but beyond a point more particles become visual noise. Research on animated edge textures notes that dense, fast particles quickly become clutter; indeed the library docs discourage speeds >0.5 (meaning more than half a link length per frame) as disorienting. For dense graphs, combine animation with **edge bundling**: group parallel flows into bundled tubes or ribbons to reduce clutter. Bundling classic papers show that grouping edges reduces crossings and makes patterns clear. In practice, you might animate a single “bundle path” instead of 10 overlapping lines.
- **Off-Screen/Overview Cues (B6):** When activity happens off-screen, provide a hint. Many UIs use a **minimap** or overview window: e.g. a small thumbnail graph in the corner that flashes where new events happen. Alternatively, edge indicators at the viewport boundary (“arrow offscreen”) or a brief ambient pulse (subtle glow along the edge closest to activity) can guide the eye. Observability dashboards typically rely on filtering or search (e.g. clicking a service highlights its entire dependency subtree), but game UIs often flash the minimap or draw a transient arrow. In absence of a direct cite, the principle is: don’t let “something happened at far right” go unseen. Use either an overlaid arrow/minimap ping or a fading highlight at the screen edge to draw attention back to that region.

# Agent Attention & Causality (C7–C8)

- **Debugging/Trace UIs:** Debugger or agent-trace visualizers usually highlight the execution path in sequence. For example, the VizLang tool “lights up” nodes as they run. Users saw each node pulse or brighten in order, making the path explicit: “watch nodes light up as they execute”. Similarly, many plan-execution interfaces display steps in a list: completed steps get checkmarks and the current step pulses with a progress indicator. These give clear before/after context (each node is marked done, next node is pulsing). That contrasts with random blinking: consistent highlighting along a path, with direction, is perceived as causal sequence.
- **Encoding Sequence on Map:** To show an agent’s navigation over time, consider drawing _trails_ or _breadcrumbs_. For instance, leave a faint line along the edges just traversed (a “comet tail” that fades). A well-known design pattern (Fuselab “plan-and-execute”) shows completed actions as checked steps and the active one with an animated indicator, which in a graph context could translate to numbered badges or glow pulses on nodes in order. Numbering (1,2,3…) or time labels on newly activated nodes is another option. While formal studies are limited, designers often note that continuous trails and anchors (checkmarks, arrows) help users follow sequence more than flashing. In short, build a clear path: e.g. animate edge arrow + fade out older trail, or keep small numbered labels next to nodes. (For example, a workflow UI might draw concentric rings revealing dependencies in BFS waves, but the key is visual continuity from A→B→C.)

# Change Impact Visualization (D9–D10)

- **Impact Graphs & Diff Tools:** Common tools highlight change impact on dependency graphs. VS Code’s **Dependency Impact Visualizer** does this: it draws a graph of impacted symbols and “shows how changes to functions/classes ripple through the codebase”. Diff tools themselves (Git, IDEs) mostly use text diffs, but modern IDEs also offer structural previews (e.g. showing renamed references in a sidebar). In a node-link context, one could color newly added/modified nodes differently. Some systems overlay “blast radius” shading: e.g. green for local change, yellow for indirect deps, red for failed/conflicts. The UI should distinguish _staged vs. applied vs. cascaded vs. failed_: for example, during a planned patch use a transient color (orange outline) on nodes to be edited, turning green as they apply, and red if an update fails. One can also use shapes or icons: e.g. an “error” badge for failed nodes. (No single canonical pattern – but emphasize layering: current edit highlights vs. ripple effects.)
- **Propagation Animation (D10):** To illustrate a change rippling through dependencies, a _wave/ripple metaphor_ works well. For example, animate dependent nodes one “ring” at a time: first highlight direct neighbors of the changed node, then neighbors-of-neighbors, etc., with a short delay between rings (a few tens of ms). This is like a breadth-first wavefront. Another approach is **topological order**: if the graph is a DAG, reveal nodes in that causal order. In either case, stagger the reveals in time so the eye sees a single expanding “blob” rather than random pops. IBM’s guidance on staggering also applies here: e.g. 20 ms per node gives a perceptible flow. A “ripple” animation (fading circles emanating) or a growing highlight along edges can signal propagation too, but ultimately reveal sequence step by step helps users mentally trace impact.

# Accessibility & Perception (E11–E13)

- **Colors for ~5–7 States:** Use a palette of distinct hues with good luminance contrast on dark gray. For example, pair blues with oranges, reds with teals, or purples with greens – combinations known to be colorblind-friendly. As Venngage notes, colorblind-safe schemes often use blue and red/orange contrasts. In practice, one might choose (for example) blue for “read,” green for “write,” yellow/orange for “patch,” magenta for “error,” etc. Ensure each color meets WCAG contrast on the dark background (e.g. the Carbon Dark theme uses gray ∼#161616 and recommends alert colors in bold blue/red). **Redundancy:** don’t rely on color alone. Use different _shapes_, _line styles_, or _icons_ for states. For instance, “error” nodes could have an exclamation shape or a distinct border, “write” could have a pencil icon, etc. Luminance (bright vs dim) can also encode importance – brighter glow for current focus, dimmer for stale. That way, even a color-blind user or someone with a monochrome screen can distinguish states.
- **Reduced-Motion:** Honor the user’s `prefers-reduced-motion`. If enabled, cut back or eliminate animations. For example, do not auto-move nodes; instead, simply jump to the final layout or fade one state into another. Use color/opacity changes to signal activity rather than movement. MDN advises that `prefers-reduced-motion` implies removing _non-essential_ motion (especially big pans/zooms) since they can cause discomfort. In practice, disable pulsing glows or particle flows (or make them very subtle), and rely on static highlights (e.g. a color change or blink) to indicate events. Many professional apps default to instant state changes or minimal fades when reduced-motion is on.
- **Glow/Pulse in Dark UI:** Glows and pulses can make elements “feel alive,” but in dark mode they must be subtle. Avoid pure-white or oversaturated glows (which vibrate on dark backgrounds). Instead, use tinted glows (e.g. a faint version of the node’s color) or soft drop-shadows. For example, a node in focus might emit a halo slightly brighter than background, but not full-white. Keep saturation moderate: Atmos recommends reducing color saturation by ~20 points in dark themes. Pulsing rings or expanding halos should be semi-transparent and low intensity. The goal is a gentle bloom rather than a neon sign. Subtle lens-flare or bloom filters (if available) can help a node look “active” without high contrast. In short, aim for **soft** light: for instance, a dim aura or slow radial pulse that’s just noticeable. This reads as “alive” but avoids eye-strain or garishness in the dark theme.

# Performance Techniques (F14–F15)

- **Canvas Optimization:** To sustain ~60 fps with many moving parts, minimize full redraws. Use a single `requestAnimationFrame` loop and only update what’s needed each frame. Techniques include _dirty-rect_ (redraw only changed regions) or layering. For example, put static content (background, unchanged nodes) on a separate canvas layer and do not clear it each frame. Only animate the dynamic layer (edges/pulsing nodes). Use **offscreen canvases** for repeated primitives (node shapes, icons): draw them once and then blit to the main canvas. This avoids re-running heavy draw commands. Also reuse objects/arrays for particles (object pooling) to reduce GC churn. Rounding all drawing coordinates to integers avoids sub-pixel anti-aliasing overhead.
- **React-Force-Graph (d3-force):** The library itself offers some controls. It supports **particle animation on links** (see `linkDirectionalParticles` and `linkDirectionalParticleSpeed`). You can assign each link a number of particles and a speed. Note speeds above ~0.5 (half-link-length per frame) are discouraged as they become disorienting. Custom node drawing (`nodeCanvasObject`) is powerful but expensive: it’s called for each node every frame. Use `pauseAnimation()` when there’s no interaction to freeze rendering (saves CPU). For camera moves, React-Force-Graph provides `centerAt(x,y,ms)` and `zoom(scale,ms)` methods – the third parameter is the animation duration in milliseconds, enabling smooth panning/zoom. There is no hard documented cap, but real-world experience suggests ~1–2k nodes / ~2–3k edges can still animate reasonably on modern hardware. Beyond that, performance may drop. In practice, avoid expensive per-frame work (like measuring text or recalculating layouts constantly). If needed, throttle force simulations or switch to a static layout (disable force on-the-fly updates) to keep the viewport responsive.
