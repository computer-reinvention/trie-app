# Sourcetrail post-mortem (A1)

Sourcetrail (formerly Coati) gained a cult following for its live code‐indexing and interactive graph view, but was shuttered in 2021. According to the founders, the decision was driven **not by a fatal UX flaw** but by practical and business issues. Maintaining a cross-platform, multi-language code indexer was “ambitious” and labor-intensive. After open-sourcing in 2019, development slowed as the original team moved on to other jobs and lost interest in long-term maintenance. They explicitly state developer tools are **hard to monetize**, especially with expectations of free/open‐source tools.

**What users loved:** Sourcetrail’s strengths were its _live, interactive_ code graph (in the spirit of a “data-driven IDE”), cross-language dependency insight, and fast “who-calls-whom” navigation beyond what typical IDEs offered. Users report that its visualization made unfamiliar code much easier to explore. For example, one developer noted Sourcetrail “did a great job at explaining larger code bases,” and another compared it favorably to heavy IDE features because it could rapidly reveal call graphs without digging through multiple “find references” dialogs.

**Limitations and failure modes:** However, several fatal limitations emerged. Users and commentators pointed out: (a) **High friction to adopt/setup:** Sourcetrail needed indexing and configuration per project; one user said it was “tricky to setup for a given project”. (b) **Overlap with IDEs:** Many felt that powerful IDEs (JetBrains, VSCode with language servers) already provide much of Sourcetrail’s value, so there was no pressing daily need for a separate tool. (c) **Niche use-case / limited frequency:** As noted on Hacker News, a call-graph tool is “something you would need only once or only every once in a while,” making it hard to justify a paid subscription. (d) **Monetization and marketing:** Sourcetrail never built a large enterprise customer base. Founders tried Patreon but suspected most companies would rather reimburse a license fee than a “donation”. Marketing was mostly to individual developers, not large organizations.

**Implications for a successor:** Any new code-graph navigator must reduce adoption friction (seamless integration/indexing), target contexts where devs will use it regularly, and address business model challenges (e.g. fit into existing IDE or toolchain). It should **leverage what users loved** — fast dependency scanning and explorable call graphs — but ensure it _delivers immediate value_ (e.g. faster than clicking through an IDE). Integration with IDE workflows or cloud services might be critical. The sourcetrail experience suggests there is **no inherent “graph UX” trap**, but rather the challenge is making graph-based navigation indispensable and easy to use. A successor should _not_ rely on funding through casual use — it will need strong enterprise adoption or a robust open-source community.

**Sources:** Sourcetrail developers’ own discontinuation post; user discussion on HN and Reddit.

# Obsidian Graph at Scale (A2)

The consensus among Obsidian power-users is that **the global graph view quickly becomes a “hairball” and loses utility at scale**. As one user put it: _“Local graph is useful, global graph is pretty but useless.”_. In practice, users use the global graph mainly to “find orphan notes” or get a motivational overview; daily navigation is done via search, folders/MOCs (Maps of Content), or backlinks. The “pretty but useless” complaint stems from: (1) **No inherent hierarchy or filtering:** the vanilla graph has no built-in way to show a hierarchy, namespaces, or user-defined grouping, so large vaults look like an undifferentiated blob. (2) **Lack of persistence:** moving around the graph redraws completely, so you can’t memorize positions or navigate smoothly. (3) **No search inside graph:** users have requested an ability to highlight or jump to specific notes in the graph.

Power-users mitigate these issues by heavy filtering or plugins. For example, many enable _local graph_ view scoped to the current note or apply filters to hide trivial nodes (e.g. daily notes, tags). Plugins like **Graphify**/Graph View enhancements (e.g. folder-based grouping, node coloring by tag/metadata, embedding note images as nodes) have become popular. These allow clustering of related notes and visual encoding of meaningful categories. In short, the community **likes the concept** of an interactive knowledge graph, but wants more structure and search. Suggested improvements include **folder-based clusters, color-coded groups, and graph search highlighting**.

**What to copy vs reject:** The “Obsidian feel” that’s worth replicating is the _interactive, organic look_ of the node-link graph (smooth animation, fluid layout) and the joy of exploring connections visually. But one should _deliberately reject_ a naïve full-graph display: beyond perhaps a few dozen nodes, unmanaged layouts become inscrutable. Key takeaways: always offer filtering (e.g. by type or folders), focus+context (local vs global), and search-in-graph. Representing metadata (tags, folders) visually is highly desired. Conversely, avoid fixed-limit global displays; instead load context around the user’s focus.

**Sources:** Obsidian user forum and Reddit comments summarizing why the default graph view fails in large vaults, and what plugin users actually find useful.

# Node-budget and DOI (A3)

Graph visualization literature and practice consistently show that **beyond a few hundred nodes, static node-link diagrams become hard to read**. In published studies, most user experiments on graph readability use on the order of 50–100 nodes. Practical tools likewise impose budgets: for instance, Gephi’s advice is to use _at most 1K nodes_ for force-directed layouts, beyond which convergence is too slow. Cytoscape (a biological network tool) notes that although it can _store_ millions of nodes, in practice layouts slow down past a few thousand. Graph exploration tools (e.g. Graphexp, Graphistry, Graphia) work around this by showing limited subsets: Graphia’s creators cite ~1M nodes/50k edges as already pushing a special GPU-based engine, while Graphexp (a scalable UI) “displays only a limited number of nodes, through which you can navigate to others”.

In short, **250 visible nodes** is well within the reasonable range for interactive layouts. It aligns with community practice: many graph UIs dynamically limit visible elements to the nearest few hundred around the user’s focus. Degree-of-interest (focus+context) filtering is a known technique in InfoVis (dating back to Furnas) for exactly this purpose. Some graph tools incorporate it: for example, Neo4j Bloom and KeyLines provide “lenses” or expansion controls to grow the graph from a seed node (effectively DOI-based) rather than render everything. We found no evidence that any major tool simply slaps down thousands of nodes at once in an end-user display. Instead, UX guidelines recommend stepwise exploration and aggregation (e.g. clustering or “+N” summaries) when networks are large.

**Failure modes:** Without limits, users quickly report clutter and drop in interaction performance. With focus+context filters, one risk is losing global structure (users may get tunnel vision). Also, naive DOI (salience minus distance) may hide important outliers if salience is measured only by static centrality. Tools often complement DOI with on-demand expansion (e.g. click a hidden cluster to expand it).

**Implication:** The ~250-node “flashlight” budget with DOI filtering is _consistent with real-world practice_. It should keep interaction smooth and avoid hairballs for 50k-scale graphs. We should just ensure DOI heuristics keep needed context (e.g. always show entry points/hubs). This approach is confirmed by other systems, but must be implemented carefully: e.g. allow the user to peel back aggregated “+N” nodes as needed.

**Sources:** Limits from Gephi/Cytoscape (via [30] and [33]), and Graphia commentary on 1M nodes being extreme; GraphExp mention of showing limited nodes.

# Animation timing and perception (B4)

Modern UI guidelines emphasize _smooth, purposeful motion_. Common wisdom (from UI guidelines and Vis literature) is that animations below ~100–150 ms can feel instantaneous and may not be noticed, while animations much longer than ~1000–1500 ms feel sluggish. The seminal Heer & Robertson paper on animated transitions showed that simple transitions of around 500–1000 ms help users follow changes; very fast (<100 ms) transitions risk being missed, and very slow (>2000 ms) become annoying or disruptive. Apple’s HIG and Material Design similarly recommend most UI animations in the 200–500 ms range (with key content changes around 300–500 ms). Over/undershoot (springiness) should be subtle: Apple advises that motion should be **“informative, focused, and not gratuitous”**.

Putting this together: for our graph, a **graph-reveal on load** could be ~800–1500 ms total (similar to an app startup animation). Individual transitions (e.g. adding/removing nodes) should be in the few-hundred-millisecond range. For example, hover-fade-outs might take ~200–300 ms to become legible out or in. Spring animations can use easing that overshoots slightly (like CSS “easeOutBack” or “spring” with moderate damping) so motion feels lively but settles quickly. We should avoid animations faster than ~100 ms (too hard to see) or slower than ~1.5 s (too slow to stay engaged). These ranges align with general UI guidelines.

**Sources:** Apple and Material motion principles (via [41†L255-L264][41†L273-L282]). Heer & Robertson’s findings (cited by [37]) and common UX heuristics suggest 300–800 ms transitions for state changes.

# Directed-edge encoding (B5)

Studies by Holten & van Wijk (CHI 2009, PacificVis 2011) found that **standard arrowheads are sub-optimal for dense graphs**. In controlled experiments, participants consistently performed _faster and more accurately_ with alternative cues. In particular, **tapered edges (wide-to-narrow lines)** and **intensity/gradient cues** (e.g. dark-to-light or color gradient along the edge) outperformed plain arrows. Holten (2009) reported that tapered lines (“fat-to-thin” edges) yielded the best response times compared to arrows or color cues. Later (PacificVis 2011), they introduced _animated/texture_ cues (“compressed” patterns), finding that compressed/animated flows could match or exceed arrows in legibility, especially on short edges. Users even _preferred_ the animated/textured edges in subjective feedback.

Our planned particles (animated flow) are similar in spirit to their “textured/animated” links. The studies suggest **particles/tapering are both effective** cues. If anything, the literature leans toward tapering as a robust default (since it won in early tests) and notes animation is promising but can be overkill for very short edges. In sum, adding motion (particles) should help, but a tapered line (or color gradient) is at least as legible for showing direction. I recommend starting with _edge tapering_ (possibly with subtle color gradient) as a baseline, and consider particles as an enhancement.

**Sources:** Holten & van Wijk 2009/2011 findings (tapered and gradient edges outperformed plain arrows, animated/textured cues performed well and were preferred).

# Force-graph performance in Electron (C6)

The `react-force-graph-2d` library (vasturiano) is widely used for moderate-sized graphs. Reports from the community suggest that rendering **a few hundred to a thousand nodes in a 2D canvas** is usually fine at interactive frame rates, but performance degrades significantly as counts rise into the thousands. For example, React-force-graph issues and forums note that ~10k nodes becomes sluggish unless using WebGL or heavy optimization (incidence, one Reddit user said it was “the best we found up to 10k” but recommended WebGL for larger).

In our case, since we limit to ~250 nodes and use Canvas (nodeCanvasObject) with custom drawing each frame, we should be in a safe performance regime. The **onEngineTick** spring logic adds per-frame work but D3’s physics is usually fast for small graphs. Electron itself adds some overhead (Chromium/Node), but at 250 nodes and Canvas, it should still run smoothly on modern hardware. Known caveats: bundling React-force-graph can inflate bundle size; ensure you tree-shake only needed parts, and watch out for memory leaks if you frequently re-mount the component. Some developers warn that complex per-node drawing can reduce FPS, so test early.

If we hit a performance cliff (say, if user clocks on many nodes quickly), alternatives include switching to the library’s WebGL mode (force-graph-3d on a 2D plane) or limiting details. But overall, given our strict node cap, 2D canvas is likely adequate.

**Sources:** Community performance anecdotes (React-Force-Graph GitHub, Reddit [55†L374-L384]) and known library docs note that ~1k–10k is the high end for 2D canvas before alternatives are needed.

# Layout stability (C7)

To preserve the user’s mental map, it’s crucial that nodes _don’t wildly jump_ when the visible set changes. Common strategies in dynamic graph layouts include: **seeded/incremental layout** (reuse old positions), **pinning/anchoring** key nodes, and **staged expansion**. In D3-force, one can supply fixed initial coordinates for nodes and only relax with a very small simulation “kick” when adding/removing nodes. In practice, this means: when refocusing, keep all still-visible nodes at their last screen positions (by setting `fx/fy` on their simulation), then add new nodes with a short animation. Some tools pre-compute a global layout (expensive) and then reveal subregions. Another approach is constraint-based: for example, treat “core” nodes as anchors (with high mass or pinned) so peripheral nodes settle around them.

The literature on dynamic graph drawing (e.g. _Fisher 1998_, _Dwyer et al._) emphasizes minimizing movement. In D3 specifically, you can avoid `force.restart()` and instead incrementally add forces or nodes. Vasturiano’s examples often “reuse” node objects so that the simulation carries over velocities. If the visible set is a strict subset of the full code graph, one could even pre-run a slow layout on the entire graph offline, then display only portions at a time (this would be stable but costly).

**Recommendation:** Implement an incremental force layout: when changing focus, _don’t re-randomize node positions_. Use the old positions as the starting point for the new force simulation. Optionally “freeze” high-degree hubs (like entry points) with fixed positions to serve as anchors. This should keep the graph from “popping” unexpectedly.

**Sources:** Graph drawing research on preserving the mental map (e.g. static layouts with incremental additions) and best practices like anchoring and reusing simulation state.
